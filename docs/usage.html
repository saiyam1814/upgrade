Usage

Every command's purpose, an example invocation, what the output looks like, the flags that matter. Nothing's hidden behind --help.

Quickstart

The day-1 production flow takes about 5 minutes from install to "you're upgrading."

# 1. Install — drops kubectl-upgrade on $PATH; kubectl auto-discovers it
curl -sSL https://raw.githubusercontent.com/saiyam1814/upgrade/main/install.sh | sh

# 2. Find what'll break
kubectl upgrade preflight --target v1.34

# 3. Get the cloud-CLI runbook
kubectl upgrade run plan --target v1.34

# 4. (You manually run the emitted commands.)
#    In a separate terminal, watch for stuck states:
kubectl upgrade run watch

# 5. After the upgrade finishes, verify
kubectl upgrade run verify --target v1.34

Output formats

human is the default, color-aware terminal output. Pick a structured format for CI or sharing:

FormatUse case
--format jsonPipe to jq, store as artifact, parse from scripts.
--format mdPaste into a PR description or issue.
--format sarifUpload to GitHub code scanning / GitLab Code Quality.

Multi-cluster mode

Every command that talks to a live cluster also accepts --contexts or --all-contexts for parallel fan-out. The fan-out is bounded by --parallel (default 8) and each cluster has a --timeout (default 90s) so a slow cluster never blocks the rest.

# Fan out preflight across many clusters
kubectl upgrade preflight --target v1.34 --contexts prod-east,prod-west,staging
kubectl upgrade preflight --target v1.34 --all-contexts --parallel 10
kubectl upgrade preflight --target v1.34 --all-contexts --exclude sandbox,scratch

RBAC: each context uses its own kubeconfig identity, so a context that can only see one namespace contributes only that namespace's findings. No cross-cluster privilege leakage.

Smart "next:" recommendations

Every command ends with a one-line → Next: suggestion in human-format output. The recommendation depends on what the command found.

preflight

The day-1 entry point. Runs every workload-side safety check and produces one combined report.

kubectl upgrade preflight --target v1.34

It runs, in order:

  1. scan — deprecated APIs in live cluster + Helm releases
  2. simulate — feature gates / defaults / kubelet / kernel changes between current and target
  3. addons — cert-manager / Karpenter / Istio / ArgoCD / Flux / prom-operator / Kyverno / ingress-nginx compat
  4. pdb — drain-deadlock detector
  5. volumes — PV/PVC/CSI/StorageClass safety
  6. vcluster — Tenant Cluster decision tree (if any vClusters detected)

Sample output:

▎ Preflight v1.30.4 → v1.34 ✓ scan: 21 objects walked ✓ simulate: forward changes added ✓ addons: compat checked ✓ pdb: drain feasibility checked ✓ volumes: PV/PVC/CSI checked ✓ vcluster: Tenant-Cluster gates checked Summary: 1 BLOCKER · 1 HIGH · 8 MEDIUM · 0 LOW · 0 INFO BLOCKERS ✗ PDB e2e/redis-pdb-broken ALLOWED DISRUPTIONS == 0 (will deadlock drain) OBJECT: policy/v1 PodDisruptionBudget e2e/redis-pdb-broken FIX: Increase replicas (≥ desiredHealthy + 1), OR maxUnavailable=1 HIGH ⚠ [v1.31] cgroup v1 support deprecated in kubelet

Useful flags: --skip-vcluster, --fail-on blocker.

fleet drift

The headline ops command for a Platform Engineering lead. Fans out across many contexts in parallel and shows the K8s version distribution + top issues across the entire fleet — so you walk into the change-window meeting with the number that matters.

# All contexts in your kubeconfig
kubectl upgrade fleet drift --target v1.34

# Specific clusters
kubectl upgrade fleet drift --target v1.34 --contexts prod-east,prod-west

# Big fleet, more parallelism
kubectl upgrade fleet drift --target v1.34 --all-contexts --parallel 20 --timeout 120

# Exclude sandbox/dev
kubectl upgrade fleet drift --target v1.34 --all-contexts --exclude sandbox

run plan · run watch · run verify

The conductor. Three subcommands wrap a real upgrade end-to-end.

run plan

Detects your provider (EKS, GKE, AKS, OpenShift, RKE2, k3s, Talos, kubeadm, vCluster) and emits the exact cloud-CLI commands. It does not execute them.

kubectl upgrade run plan --target v1.34
▎ Upgrade Runbook v1.30.4 → v1.34 provider: gke [1/6] Pre-flight scan → Run pre-flight to find every workload-side time bomb: $ kubectl upgrade preflight --target v1.34 [3/6] Control-plane upgrade (gke) → Verify auth + current state: $ gcloud auth list $ gcloud container clusters describe <cluster-name> --zone us-central1-a --format='value(currentMasterVersion)' → Run the control-plane upgrade: $ gcloud container clusters upgrade <cluster-name> --master --cluster-version=1.34 --zone us-central1-a

Zonal vs regional GKE clusters use the right flag automatically.

run watch

Polls the cluster on an interval and surfaces every stuck pattern as it appears. Run in a separate terminal while the upgrade is in progress.

kubectl upgrade run watch                # 30s interval, 60min cap
kubectl upgrade run watch --interval 10  # be more aggressive
kubectl upgrade run watch --stop-after 120

Stops after 3 consecutive clean cycles AND the server version matches your target — then suggests run verify.

run verify

Post-upgrade verification. Confirms server version, no NotReady nodes, no critical operator CrashLoopBackoff, no stuck states.

kubectl upgrade run verify --target v1.34
▎ Verify upgrade target v1.34 ✓ server version v1.34.5-gke.1153000 matches target ✓ no stuck states detected

fleet

vCluster Tenant Cluster wave orchestration. When the host bumps, every tenant gets a per-tenant pre-flight + ordered upgrade wave.

Status (no flags)

kubectl upgrade fleet

Lists every Tenant Cluster on this Control Plane Cluster — vCluster version, distro, topology, backing store.

Host bump check (the answer to "is my host upgrade safe for my fleet?")

kubectl upgrade fleet --host-target v1.34 --plan

For each tenant, runs the host × vCluster compat matrix:

vCluster bandSupported host K8s window
v0.20 – v0.211.26 … 1.30
v0.22 – v0.231.27 … 1.31
v0.241.28 … 1.32
v0.25 – v0.271.29 … 1.33
v0.28 – v0.301.29 … 1.34
v0.31 – v0.331.30 … 1.35
v0.34+1.31 … 1.36

Three states per tenant:

  • ✓ INFO — current vCluster supports the new host. Safe.
  • ⚠ MEDIUM — at the upper edge. Plan a tenant bump after the host.
  • ✗ BLOCKER — outside the window. Bump the tenant first; the runbook tells you the minimum vCluster version.

Wave plan + per-tenant runbook

kubectl upgrade fleet --vcluster-target v0.34.0 --plan

Emits a sequenced wave (BLOCKERs first → external-etcd → embedded-etcd → Deployment topology last). Each step:

[1/2] Tenant Cluster team-a/vc-team-a → Snapshot before mutating: $ vcluster snapshot create vc-team-a -n team-a oci://<reg>/<repo>:vc-team-a-pre-upgrade → Upgrade Tenant Cluster: $ helm upgrade --install vc-team-a vcluster --repo https://charts.loft.sh --version v0.34.0 -n team-a --reuse-values

scan

Deprecated APIs in live cluster + Helm releases + raw manifests.

# Live cluster + Helm release secrets (default)
kubectl upgrade scan --target v1.34

# Just static manifests
kubectl upgrade scan --target v1.34 --source manifests --path ./k8s/

# Combine sources
kubectl upgrade scan --target v1.34 --source live,helm,manifests --path ./k8s/

# CI gate
kubectl upgrade scan --target v1.34 --fail-on blocker

Helm-release scanning catches "shadow APIs" — objects baked into a release whose API the apiserver may no longer serve. Helm v3 stores one Secret per release revision; scan dedupes to the latest revision per release.

simulate

Forward simulator: feature gates / defaults / kubelet flags / kernel reqs that activate between two minors.

kubectl upgrade simulate --from v1.31 --to v1.34

Surfaces things scan can't: PSA enforcement defaults flipping, cgroup v1 deprecation/removal, kubelet flag removals, scheduler queueing-hints, in-tree → CSI migration, etc. Curated through K8s 1.36.

pdb

PodDisruptionBudgets that would deadlock a node drain.

kubectl upgrade pdb

Flags every PDB where ALLOWED DISRUPTIONS == 0 — the canonical "stuck upgrade" pattern. Also flags PDBs whose selector matches zero pods (configuration drift).

A PDB with minAvailable: 1 and replicas: 1 is the most common upgrade-blocker. Either add a second replica, set maxUnavailable: 1, or accept downtime.

addons

Compatibility check for installed third-party controllers against a target K8s version.

kubectl upgrade addons --target v1.34

Detects: cert-manager, Karpenter, Istio, ArgoCD, Flux, prometheus-operator, Kyverno, ingress-nginx. For each: emits the supported K8s window and flags incompatible target combos.

ingress-nginx is a special case — the project retired 2026-03-31. The check always emits a HIGH advisory regardless of target.

volumes

PV / PVC / CSI / StorageClass safety surface.

kubectl upgrade volumes --target v1.34

What it catches:

  • StorageClasses on in-tree provisioners whose CSI driver isn't installed (in-tree removed in 1.31+)
  • Pending PVCs that will block pod reschedule on drain
  • Deployments mounting PVCs (wrong primitive — use StatefulSet)
  • StatefulSets with replicas=1 (outage window during node drain)
  • ReadWriteMany volumes on cloud classes mid-removal
  • Released PVs that should be reclaimed
  • Missing VolumeSnapshotClass (needed for safe pre-upgrade snapshots)

vcluster

Per-Tenant-Cluster decision tree. Where fleet works at the herd level, vcluster drills into each tenant's product-version upgrade path.

kubectl upgrade vcluster --target v0.34.0
kubectl upgrade vcluster --explain   # dump the entire decision tree

Gates checked:

  1. Pre-v0.20 sources need vcluster convert config (values.yaml → vcluster.yaml)
  2. Distro removal: k3s removed v0.33, k0s v0.26, eks v0.20
  3. Skip-minor refusal — emits a chained per-minor plan
  4. etcd 3.5 → 3.6 transition crossing v0.29
  5. Topology safety (Deployment + non-external state = data loss)
  6. Backup reminder (downgrades are unsupported after v0.20)
  7. Sleep-state advisory (Virtual Control Plane must be awake to snapshot)

unstick

Stuck-state recovery toolkit. Run when an upgrade has stalled.

# Report only (read-only, default)
kubectl upgrade unstick

# Apply the safe class of fixes (uncordon nodes only)
kubectl upgrade unstick --auto-fix --execute

Detects:

  • Cordoned nodes left over post-drain
  • NotReady nodes (CNI / kubelet)
  • Pods stuck Terminating or Pending past --stuck-min minutes
  • Operator Pods in CrashLoopBackoff (cert-manager, ArgoCD, Karpenter, …)
  • PDB-blocked evictions in recent events
  • Webhooks with failurePolicy=Fail (deadlock risk)
  • Namespaces stuck Terminating (finalizer deadlock)

--auto-fix --execute only applies safe fixes — uncordoning nodes is the only one in v0.1. Risky fixes (force-delete pods, patch webhooks, remove namespace finalizers) emit the command for you to run.

plan

Chained one-minor-at-a-time upgrade path.

kubectl upgrade plan --from v1.30 --to v1.34
Chained upgrade plan: v1.30 → v1.31 → v1.32 → v1.33 → v1.34 Hops: 4 Per-hop checklist: 1. Take a fresh etcd snapshot / Velero backup. 2. Run `upgrade scan --target <next-hop>`. Resolve all BLOCKERS. 3. Run `upgrade addons --target <next-hop>`. Bump incompatible operators FIRST. 4. Run `upgrade pdb`. Fix ALLOWED DISRUPTIONS == 0 PDBs. 5. Pause GitOps reconciliation (ArgoCD auto-sync, Flux suspend). 6. Upgrade control plane. 7. Upgrade nodes (rolling, max-surge ≥ 1). 8. Verify workloads, resume GitOps, re-snapshot as new baseline.

report

Aggregated combined report. Same shape as preflight, but adds a chosen output format and fail-on gate.

kubectl upgrade report --target v1.34 --format md -o upgrade-readiness.md

tui

Interactive visual upgrade dashboard (bubbletea).

kubectl upgrade tui --target v1.34

Side-by-side dashboard with steps on the left, per-step details / findings / commands on the right.

CI / CD recipe

Gate every PR on a clean preflight against a target version.

GitHub Actions

name: K8s upgrade pre-flight
on:
  pull_request:
    paths: ['**/*.yaml', '**/*.yml']

jobs:
  preflight:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install kubectl-upgrade
        run: |
          curl -sSL https://raw.githubusercontent.com/saiyam1814/upgrade/main/install.sh | sh
      - name: Scan manifests for deprecated APIs
        run: |
          kubectl-upgrade scan \
            --target v1.34 \
            --source manifests --path ./k8s/ \
            --fail-on blocker \
            --format sarif -o upgrade.sarif
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: upgrade.sarif

For full pre-flight against the live cluster (e.g. a nightly job):

      - name: Live preflight
        run: |
          kubectl-upgrade preflight \
            --target v1.34 \
            --fail-on high \
            --format md -o readiness.md

GitLab / generic

# Anywhere with kubectl access:
kubectl-upgrade preflight --target v1.34 --fail-on blocker

Exit codes: 0 = clean, 2 = at least one finding ≥ --fail-on.

Safety contract

Every command is read-only by default. The single mutating path (unstick --auto-fix) requires both --execute and a per-action [y/N] confirm. The full safety contract is at SAFETY.md.

kubectl-upgrade NEVER runs cloud CLIs (aws / gcloud / az / talosctl / kubeadm) for you. It emits the exact command. You execute. The blast radius of a control-plane upgrade is yours, not ours.
